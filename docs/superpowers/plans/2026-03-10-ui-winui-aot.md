# ISynergy.Framework.UI.WinUI AOT Compatibility Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `ISynergy.Framework.UI.WinUI` NativeAOT-compatible for Windows App SDK NativeAOT publishing by eliminating runtime assembly scanning, reflection-based property introspection, and `Type.GetType(string)` calls, and replacing them with generated compile-time dispatch tables.

**Architecture:** WinUI 3 on Windows App SDK supports NativeAOT since SDK 1.5. The library's critical AOT blockers are: (1) the shared `GetRelatedViewType` assembly scan (fixed by base UI source generator), (2) `GetProperties`-based command introspection in both `NavigationService` and `ViewModelExtensions`, (3) `Type.GetType(string)` in `EnumConverters.EnumToBooleanConverter`, and (4) `Enum.GetValues(runtimeType)` in `EnumToArrayConverter` and `EnumKeyValueCollection`. No new generator is needed beyond the base plan; fixes are annotation-based or pattern replacements.

**Tech Stack:** .NET 10 / Windows App SDK / WinUI 3 NativeAOT, `ISynergy.Framework.UI.SourceGenerator` from base plan, C# 14.

## Implementation Steps

- [ ] **Step 1 — Consume generated registration in `WindowsAppBuilderExtensions`**

  Replace:
  ```csharp
  services.RegisterAssemblies(assembly, assemblyFilter);
  ```
  With:
  ```csharp
  services.AddUITypes(); // generated by ISynergy.Framework.UI.SourceGenerator
  ```

- [ ] **Step 2 — Fix `NavigationExtensions.CreatePage<TViewModel>` view-type resolution**

  Use the generated `ViewModelViewMap.Map` dictionary before falling back to `GetRelatedViewType`:
  ```csharp
  var viewName = typeof(TViewModel).GetRelatedView();
  var viewType = ViewModelViewMap.Map.TryGetValue(viewName, out var t) ? t
      : viewName.GetRelatedViewType(); // [RequiresUnreferencedCode] fallback
  ```

- [ ] **Step 3 — Fix `HasRunningCommands` reflection in `NavigationService`**

  Replace `GetProperties`-based command introspection with an interface check:
  ```csharp
  private static bool HasRunningCommands(IViewModel viewModel)
  {
      if (viewModel is ICommandProvider provider)
          return provider.GetCommands().OfType<IAsyncRelayCommand>().Any(c => c.IsRunning);
      return false; // AOT-safe: no reflection
  }
  ```
  Add `[RequiresUnreferencedCode]` to the old pattern as a build guard. The `ICommandProvider` interface is defined in the base MVVM project (see base UI plan, Step 5).

- [ ] **Step 4 — Fix `EnumToBooleanConverter` — `Type.GetType(string)` pattern**

  The `EnumType` string property is inherently AOT-hostile because the type name is specified in XAML markup. Two options:

  **Option A (recommended):** Change `EnumType` from `string` to `Type`:
  ```csharp
  public Type? EnumType { get; set; }  // Was: public string EnumType
  ```
  XAML callers use `x:Type` syntax: `<converters:EnumToBooleanConverter EnumType="{x:Type local:MyEnum}" />`. This makes the type statically known to the linker.

  **Option B:** Add `[DynamicDependency(DynamicallyAccessedMemberTypes.All, typeof(MyEnum))]` at each call site — impractical for a library converter.

  Implement **Option A** and mark the old string-based path `[RequiresUnreferencedCode]` for a transition period.

- [ ] **Step 5 — Fix `EnumToStringConverter` — `Type.GetType(string)` pattern**

  Same as Step 4: change the converter to accept `Type parameter` in XAML or annotate with `[RequiresUnreferencedCode]`.

  For `EnumToStringConverter.Convert` line 150–151, the `parameter` is passed at runtime from XAML binding. If the parameter is a string type name, it is AOT-hostile. Recommend annotating `Convert` with `[RequiresUnreferencedCode]` and documenting that the string-type-name pattern is not AOT-safe. Provide an AOT-safe overload using `Enum.Parse<TEnum>`.

- [ ] **Step 6 — Fix `EnumToArrayConverter.Convert` and `EnumKeyValueCollection` — `Enum.GetValues(runtimeType)`**

  `Enum.GetValues(Type)` requires the enum type's members to be preserved. For NativeAOT, add:
  ```csharp
  [RequiresDynamicCode("Enum.GetValues requires dynamic code generation.")]
  [RequiresUnreferencedCode("Enum type members may be trimmed.")]
  ```
  to `EnumToArrayConverter.Convert` and `EnumKeyValueCollection.ProvideValue`.

  Provide AOT-safe generic alternatives:
  ```csharp
  public static IEnumerable<KeyValuePair<int, string>> GetEnumValues<TEnum>() where TEnum : struct, Enum
      => Enum.GetValues<TEnum>().Select(e => new KeyValuePair<int, string>((int)(object)e, GetDescription(e)));
  ```
  `Enum.GetValues<TEnum>()` (generic overload, .NET 5+) is AOT-safe.

- [ ] **Step 7 — Fix `EnumCollection.ProvideValue` — `EnumType.ToList()` call**

  Verify that `ToList()` extension on `Type` (in `ISynergy.Framework.Core.Extensions`) uses `Enum.GetValues`. If so, apply the same `[RequiresUnreferencedCode]` annotation and encourage consumers to use a generic alternative.

- [ ] **Step 8 — Fix `DialogService.ShowDialogAsync<TEntity>(IWindow, ...)` runtime `GetType()`**

  Line 336: `_scopedContextService.GetRequiredService(window.GetType())` — annotate with `[RequiresUnreferencedCode]`. Document that `ShowDialogAsync<TWindow, TViewModel, TEntity>()` is the AOT-safe overload.

- [ ] **Step 9 — Enable NativeAOT analysis in project file**

  In `ISynergy.Framework.UI.WinUI.csproj` add:
  ```xml
  <IsTrimmable>true</IsTrimmable>
  <EnableTrimAnalyzer>true</EnableTrimAnalyzer>
  ```
  Verify by publishing a test WinUI host with `<PublishAot>true</PublishAot>` and confirming no `IL3xxx`/`IL2xxx` errors originate from this assembly.

- [ ] **Step 10 — Write unit tests**

  - Test `EnumToBooleanConverter` with `Type` property (Step 4)
  - Test `HasRunningCommands` delegates to `ICommandProvider`
  - Test `NavigationExtensions.CreatePage` uses generated map

## Notes

- **WinUI 3 NativeAOT** is supported on Windows App SDK 1.5+. Publishing requires `<PublishAot>true</PublishAot>` in the app project (not the library).
- The XAML compiler (XBF) preserves type references for `x:Type` usages, so changing converters from `string` to `Type` properties is XAML-linker safe.
- `DependencyObject.FindDescendant<Frame>()` in NavigationService is a generic visual-tree walk; generic type parameter is known at compile time — AOT-safe.
- `_backStackChanged?.GetInvocationList()` (in NavigationService) uses delegate reflection; `GetInvocationList()` is preserved by NativeAOT but may produce `IL3050` warnings. Annotate with `[RequiresDynamicCode]` or replace with a `List<EventHandler>` backing store.

## Migration Guide

WinUI 3 NativeAOT support requires switching from `RegisterAssemblies` to `AddUITypes()`, updating enum converters from string-based to `Type`-based `EnumType` properties, and implementing `ICommandProvider` on ViewModels.

### Before

```csharp
// WindowsAppBuilderExtensions — reflection-based scan
services.RegisterAssemblies(assembly, assemblyFilter);

// XAML — string-based enum type (AOT-hostile)
<converters:EnumToBooleanConverter EnumType="MyApp.MyStatus" />
```

### After

```csharp
// WindowsAppBuilderExtensions — source-generator-emitted registration
services.AddUITypes(); // generated by ISynergy.Framework.UI.SourceGenerator

// XAML — x:Type-based enum type (AOT-safe, linker can see the reference)
<converters:EnumToBooleanConverter EnumType="{x:Type local:MyStatus}" />
```

### Steps to Migrate

1. Replace `services.RegisterAssemblies(assembly, assemblyFilter)` with `services.AddUITypes()`.
2. In XAML, change `EnumType="Namespace.EnumName"` to `EnumType="{x:Type local:EnumName}"` for all uses of `EnumToBooleanConverter` and `EnumToStringConverter`.
3. Implement `ICommandProvider` on your ViewModels; the `NavigationService.HasRunningCommands` method will use this interface instead of property reflection.
4. Replace `EnumToArrayConverter` usages in XAML with a generic `Enum.GetValues<TEnum>()` call in the ViewModel if the enum type is statically known.
5. For `DialogService.ShowDialogAsync<TEntity>(IWindow, ...)`, prefer the strongly-typed `ShowDialogAsync<TWindow, TViewModel, TEntity>()` overload to avoid the runtime `GetType()` path.
