# ISynergy.Framework.UI.Maui AOT Compatibility Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `ISynergy.Framework.UI.Maui` AOT-compatible for iOS/Android MAUI linker (IL Trimming) and future NativeAOT targets by eliminating all runtime type scanning and reflection-based navigation, replacing them with the compile-time generated registration and view-map from `ISynergy.Framework.UI.SourceGenerator`.

**Architecture:** The MAUI library has two AOT blocker clusters: (1) the inherited `RegisterAssemblies` + `GetRelatedViewType` chain (shared with all platforms, addressed by the base UI source generator plan), and (2) MAUI-specific runtime `GetType().Namespace` string comparison in `ApplicationExtensions.SetApplicationColor` (harmless but worth noting) and `GetType().Name` string comparisons in `NavigationExtensions`. After the base generator fix, the remaining MAUI-specific work is annotation-only — no additional source generator is needed.

**Tech Stack:** .NET 10 MAUI (iOS/Android/Windows/macOS), .NET MAUI linker (MSBuild `PublishTrimmed=true`), `ISynergy.Framework.UI.SourceGenerator` from base plan.

## Implementation Steps

- [ ] **Step 1 — Consume generated registration in `MauiAppBuilderExtensions.ConfigureServices`**

  Replace:
  ```csharp
  services.RegisterAssemblies(assembly, assemblyFilter);
  ```
  With:
  ```csharp
  // AOT-safe: generated by ISynergy.Framework.UI.SourceGenerator at build time
  services.AddUITypes();
  ```
  Keep the old overload as `[RequiresUnreferencedCode]`-annotated for fallback.

- [ ] **Step 2 — Update `NavigationExtensions.CreatePage<TViewModel>` to use generated view map**

  Replace the `view.GetRelatedViewType()` call in `NavigationExtensions.cs` (lines 31–32) with:
  ```csharp
  var viewType = ViewModelViewMap.Map.TryGetValue(view, out var t) ? t
      : view.GetRelatedViewType(); // fallback for non-AOT scenarios
  ```
  The generated `ViewModelViewMap.Map` (from the base plan generator) eliminates the runtime assembly scan on the hot path.

- [ ] **Step 3 — Fix `GetType().Name` comparisons in `NavigationExtensions`**

  Replace `GetType().Name` equality checks (lines 68, 88) with direct type equality:
  ```csharp
  // Before
  !Application.Current.Windows[0].GetType().Name.Equals(page.GetType().Name)
  // After
  Application.Current.Windows[0].GetType() != page.GetType()
  ```
  This avoids string allocation and is fully AOT-safe.

- [ ] **Step 4 — Fix `DialogService.ShowDialogAsync<TEntity>(IWindow, ...)` runtime `GetType()` call**

  Line 593: `_scopedContextService.GetRequiredService(window.GetType())` resolves the dialog window by runtime type. This pattern is needed because the caller passes an `IWindow` interface, not a concrete generic type. Fix options:
  - Option A: Change the method signature to `ShowDialogAsync<TEntity, TWindow>(TWindow window, ...)` to make the type statically known
  - Option B: Add `[RequiresUnreferencedCode]` to this specific overload and document that callers should prefer the generic `ShowDialogAsync<TWindow, TViewModel, TEntity>()` overload

  Recommend **Option B** for minimal breaking change. Add `[RequiresUnreferencedCode("Use ShowDialogAsync<TWindow,TViewModel,TEntity>() for AOT-safe dialog display.")]`.

- [ ] **Step 5 — Fix `ApplicationExtensions` theme namespace comparison**

  In `SetApplicationColor`, replace:
  ```csharp
  dict.GetType().Namespace == "ISynergy.Framework.UI.Resources.Styles.Themes"
  ```
  With a marker interface check:
  ```csharp
  dict is IThemeResourceDictionary
  ```
  Add `IThemeResourceDictionary` marker interface to `ISynergy.Framework.UI.Maui` and implement it on all theme `ResourceDictionary` subclasses (`Themeffb900`, etc.).

- [ ] **Step 6 — Audit `BaseShellViewModel` for reflection**

  Read `src/ISynergy.Framework.UI.Maui/ViewModels/Base/BaseShellViewModel.cs`. If it uses `GetType().GetProperties()` or similar patterns, annotate with `[RequiresUnreferencedCode]` or replace with interface-based access.

- [ ] **Step 7 — Enable trimmer analysis in project file**

  In `ISynergy.Framework.UI.Maui.csproj` add:
  ```xml
  <IsTrimmable>true</IsTrimmable>
  <EnableTrimAnalyzer>true</EnableTrimAnalyzer>
  ```
  Build for Android (`dotnet build -f net10.0-android`) and confirm no `IL2xxx` warnings.

- [ ] **Step 8 — Write tests**

  - Test that `NavigationExtensions.CreatePage<TViewModel>` uses the generated map when available
  - Test that `ApplicationExtensions.SetApplicationColor` applies the correct theme by type (not namespace string)
  - Verify `DialogService` overloads with `[RequiresUnreferencedCode]` surface the annotation at call sites in test projects

## Notes

- **MAUI iOS/macOS** uses the Mono AOT compiler (not NativeAOT), which has `GetType()` available at runtime but respects linker trimming. The critical concern is the linker removing types not statically referenced — not strict NativeAOT semantics.
- **MAUI Android** uses the Mono AOT compiler as well. Same trimming concern.
- **MAUI Windows** runs on full .NET 10 WinUI, where trimming is optional. The fixes are still beneficial for correctness.
- The `AppDomainUnloadedException` usage in `DialogService.IsCriticalDialogException` is fine — it is a type reference, not assembly scanning.
- `Task.Run(() => Console.WriteLine(...))` in fallback notification paths creates thread-pool work items; this is trimming-safe.

## Migration Guide

For MAUI applications, `RegisterAssemblies` is replaced by `AddUITypes()`, navigation uses the generated view-map, and theme detection switches from a namespace string comparison to a marker interface check.

### Before

```csharp
// MauiProgram.cs — reflection-based assembly scan
services.RegisterAssemblies(
    Assembly.GetAssembly(typeof(App)),
    name => name.Name?.StartsWith("MyApp") == true);

// ApplicationExtensions — fragile namespace string check
// (internal; dict.GetType().Namespace == "ISynergy.Framework.UI.Resources.Styles.Themes")
```

### After

```csharp
// MauiProgram.cs — source-generator-emitted registration
services.AddUITypes(); // generated by ISynergy.Framework.UI.SourceGenerator

// ApplicationExtensions — marker interface check (AOT-safe)
// (internal; dict is IThemeResourceDictionary)
```

### Steps to Migrate

1. Remove the `RegisterAssemblies(assembly, assemblyFilter)` call from `MauiProgram.cs` and replace it with `services.AddUITypes()`.
2. Update `NavigationExtensions.CreatePage<TViewModel>` to use the generated `ViewModelViewMap.Map` dictionary for view-type resolution.
3. Replace `GetType().Name` equality checks in navigation with direct `GetType()` equality: `Application.Current.Windows[0].GetType() != page.GetType()`.
4. For `DialogService.ShowDialogAsync<TEntity>(IWindow, ...)`, add `[RequiresUnreferencedCode]` suppression at call sites or switch to the strongly-typed generic overload.
5. Implement `IThemeResourceDictionary` on all theme `ResourceDictionary` subclasses in your application so that `SetApplicationColor` can identify theme dictionaries without a namespace string comparison.
