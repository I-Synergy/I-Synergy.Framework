# ISynergy.Framework.UI.UWP AOT Compatibility Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Improve `ISynergy.Framework.UI.UWP` compatibility with .NET Native (UWP's ahead-of-time compilation system) by eliminating patterns that are known to fail under `.NET Native` toolchain, primarily `AppDomain.CurrentDomain.GetAssemblies()` scanning, runtime `Type.GetType(string)`, and property reflection.

**Architecture:** UWP uses **.NET Native** (not the .NET NativeAOT toolchain). .NET Native has been the UWP AOT mechanism since Windows 8.1 and is distinct from `<PublishAot>true</PublishAot>`. It performs whole-program optimization and requires that reflected types be declared in a `Default.rd.xml` runtime directives file. This plan replaces the most problematic reflection patterns with the compile-time generated registration table and annotates remaining reflection with `[RequiresUnreferencedCode]` (recognized by the .NET Native compiler's analysis tooling).

**Tech Stack:** .NET 10-windows10.0.26100.0 UWP, .NET Native toolchain, `ISynergy.Framework.UI.SourceGenerator` from base plan.

---

## Platform Limitation Statement

> UWP uses **.NET Native** (not NativeAOT). The `.NET Native` toolchain has its own reflection policy driven by `Default.rd.xml`. While full `PublishAot` is not applicable to UWP, the patterns fixed in this plan are equally harmful under .NET Native because they attempt to enumerate assemblies or look up types by name at runtime — operations that .NET Native's linker removes unless declared in `rd.xml`.

## Notes

- **UWP and NativeAOT**: UWP uses `.NET Native` (the legacy AOT compiler for UWP apps), not the .NET 5+ NativeAOT toolchain. They share similar principles but have different tooling. The `<PublishAot>true</PublishAot>` MSBuild property does NOT apply to UWP.
- **`AppDomain.CurrentDomain.GetAssemblies()`**: Under .NET Native for UWP, `AppDomain` is implemented but `GetAssemblies()` returns only assemblies the .NET Native linker has included. If views/viewmodels are in assemblies the linker considers unreferenced, they will not be discoverable at runtime. This makes the assembly-scan pattern doubly broken under .NET Native.
- **`Default.rd.xml`**: All types accessed via reflection in UWP .NET Native apps must be declared in the runtime directives file. Consuming apps must add entries for any ViewModel/View types they define, regardless of this library's fixes, unless the library registers them at compile time via the source generator.
- The UWP `NavigationService` is structurally identical to the WinUI version (same `HasRunningCommands` pattern, same `GetRelatedViewType` chain). The fixes are identical.
- The UWP `Application.cs` structure mirrors WinUI; the same `_backStackChanged?.GetInvocationList()` consideration applies.

## Migration Guide

UWP uses `.NET Native`, not NativeAOT. The `RegisterAssemblies` call is replaced by `AddUITypes()`, enum converters are updated to use `Type`-based properties, and `HasRunningCommands` is replaced by `ICommandProvider`. Additionally, consuming UWP apps must maintain a `Default.rd.xml` runtime directives file for any remaining reflected types.

### Before

```csharp
// WindowsAppBuilderExtensions — reflection-based scan
services.RegisterAssemblies(assembly, assemblyFilter);

// XAML — string-based enum type (not .NET Native safe)
<converters:EnumToBooleanConverter EnumType="MyApp.MyStatus" />
```

### After

```csharp
// WindowsAppBuilderExtensions — source-generator-emitted registration
services.AddUITypes(); // generated by ISynergy.Framework.UI.SourceGenerator

// XAML — x:Type-based enum type (.NET Native linker can see the reference)
<converters:EnumToBooleanConverter EnumType="{x:Type local:MyStatus}" />
```

### Steps to Migrate

1. Replace `services.RegisterAssemblies(assembly, assemblyFilter)` with `services.AddUITypes()`.
2. In XAML, change all `EnumType="Namespace.EnumName"` string values to `EnumType="{x:Type local:EnumName}"`.
3. Implement `ICommandProvider` on your ViewModels so `HasRunningCommands` works without reflection.
4. Add a `Default.rd.xml` runtime directives file to your UWP application project. At minimum, include:
   ```xml
   <Namespace Name="ISynergy.Framework.UI.Services" Serialize="All" Browse="All" />
   <Namespace Name="ISynergy.Framework.UI.Converters" Serialize="All" Browse="All" />
   ```
5. For any ViewModel/View types that cannot be discovered via the source generator (e.g., in separate assemblies), add `<Type Name="MyApp.ViewModels.UserViewModel" Browse="Required All" />` entries in `Default.rd.xml`.
