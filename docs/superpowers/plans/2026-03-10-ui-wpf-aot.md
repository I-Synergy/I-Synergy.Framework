# ISynergy.Framework.UI.WPF AOT Compatibility Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Improve `ISynergy.Framework.UI.WPF` trimming-safety and reduce reliance on reflection patterns, within the constraints of WPF's own trimming limitations, so that consuming applications can enable `PublishTrimmed=true` with fewer suppressions.

**Architecture:** WPF on .NET 10 does **not** support NativeAOT. Microsoft's official position is that WPF remains reflection-heavy due to XAML binding, `DependencyProperty` registration, and the markup compiler. However, the .NET 10 WPF SDK does support `PublishTrimmed=true` with warnings. This plan annotates the framework library code for trim-safety, replaces avoidable reflection patterns, and provides an AOT-safe registration path — even if the WPF runtime itself requires trimming suppressions. The plan does **not** promise NativeAOT compatibility for WPF.

**Tech Stack:** .NET 10-windows WPF, `PublishTrimmed=true` (linker/trimmer, not NativeAOT), `ISynergy.Framework.UI.SourceGenerator` from base plan.

---

## Platform Limitation Statement

> WPF does NOT support NativeAOT (`<PublishAot>true</PublishAot>`). This plan targets **trim-safe library code** only. Trimmer warnings (`IL2xxx`) emitted by WPF's own infrastructure are out of scope and should be suppressed at the application level with `<TrimmerRootDescriptor>` or `$(NoWarn)`.

## Notes

- **WPF and NativeAOT**: As of .NET 10, WPF explicitly excludes itself from NativeAOT support. The `WindowsDesktop` workload ships WPF with a `.runtimeconfig.json` entry `"configProperties": { "System.Windows.Application.EnableNativeAot": false }` (or equivalent). Do not attempt to enable NativeAOT in WPF apps — it will fail at the WPF runtime level regardless of library code.
- **Trimming WPF apps**: `PublishTrimmed=true` is possible for WPF apps since .NET 6 with `<SuppressTrimAnalysisWarnings>true</SuppressTrimAnalysisWarnings>` at the app level. The goal of this plan is to eliminate suppressions that originate from _our_ library code.
- The WPF `DependencyObjectExtensions.FindDescendant<T>()` used in `NavigationService` is a generic visual-tree traversal; `T` is statically known — trim-safe.
- The WPF `Frame.Content` assignment pattern is an `object` property; the linker preserves `object` members. No issue.
- `_backStackChanged?.GetInvocationList()` in NavigationService: `GetInvocationList()` is available at runtime in trimmed apps but generates `IL3050` for NativeAOT. Since WPF is not NativeAOT, this is low priority — annotate for completeness.

## Migration Guide

WPF does not support NativeAOT. This plan targets `PublishTrimmed=true` only. The `RegisterAssemblies` call is replaced by `AddUITypes()` from the base UI source generator, and `HasRunningCommands` reflection is replaced by an `ICommandProvider` interface check. **Do not set `<PublishAot>true</PublishAot>` in WPF apps.**

### Before

```csharp
// WPF App.xaml.cs — reflection-based assembly scan
services.RegisterAssemblies(
    Assembly.GetExecutingAssembly(),
    name => name.Name?.StartsWith("MyApp") == true);
```

### After

```csharp
// WPF App.xaml.cs — source-generator-emitted registration
services.AddUITypes(); // generated by ISynergy.Framework.UI.SourceGenerator
```

### Steps to Migrate

1. Remove the `RegisterAssemblies(Assembly, ...)` call from your WPF application startup and replace it with `services.AddUITypes()`.
2. Implement `ICommandProvider` on your ViewModels to expose the list of commands — this allows `HasRunningCommands` to work without `GetProperties` reflection.
3. In the WPF application project file, add `<IsTrimmable>true</IsTrimmable>` and `<EnableTrimAnalyzer>true</EnableTrimAnalyzer>` (do NOT add `<PublishAot>true</PublishAot>`).
4. Publish with `dotnet publish -p:PublishTrimmed=true -f net10.0-windows` and verify that no IL2xxx warnings originate from `ISynergy.Framework.UI.WPF` assemblies.
5. Any WPF infrastructure warnings (from WPF's own XAML binding system) may be suppressed with `<SuppressTrimAnalysisWarnings>true</SuppressTrimAnalysisWarnings>` at the application level if needed.
