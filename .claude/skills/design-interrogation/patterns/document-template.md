# Architecture Document Template

Every architecture document generated by the design-interrogation skill MUST follow this template.
Sections marked **[MANDATORY]** must always be present.
Sections marked **[CONDITIONAL]** are required only when the relevant technology is chosen.

Do not omit mandatory sections — fill them even when the answer is brief.
Do not include SQL DDL, hardcoded connection strings, MediatR references, or `DateTime` types anywhere.

### 4.3 Projections [CONDITIONAL — if Event Sourcing chosen]

**Storage:** Typed projection tables in domain schemas
**Implementation Pattern:** EF Core `IEntityTypeConfiguration<T>` — NO SQL DDL

**Entity Class Example:**
```csharp
public sealed class {Name}Projection
{
    public string TenantId { get; set; } = null!;
    public Guid {AggregateId} { get; set; }
    public string {Property} { get; set; } = null!;
    public DateTimeOffset {TimestampProperty} { get; set; }
    public long Version { get; set; }
    public DateTimeOffset LastUpdated { get; set; }
}
```

**Configuration Class Example:**
```csharp
public sealed class {Name}ProjectionConfiguration
    : IEntityTypeConfiguration<{Name}Projection>
{
    public void Configure(EntityTypeBuilder<{Name}Projection> builder)
    {
        builder.ToTable("{name}_projection", "{schema}");
        builder.HasKey(e => new { e.TenantId, e.{AggregateId} });
        builder.Property(e => e.{Property})
            .HasColumnName("{property}")
            .HasMaxLength({n})
            .IsRequired();
        builder.HasIndex(e => new { e.TenantId, e.{IndexField} })
            .HasDatabaseName("ix_{name}_{index_field}");
    }
}
```

**Rebuild Strategy:** `IProjector` pattern — replay events ordered by `RowKey` via `SequenceNumberFormatter.Parse()`

**Pattern References:**
- See `.github/instructions/event-sourcing-projections.instructions.md` — IProjector, IProjectionReader\<T\>, naming rules
- See `.github/instructions/entityframework-core.instructions.md` — IEntityTypeConfiguration rules
- See `.github/instructions/naming-conventions.instructions.md` — `{Domain}DataContext` (NOT DbContext)

### 4.4 CQRS Implementation [MANDATORY]

**Pattern:** Direct handler injection — NO MediatR

**Command definition:**
```csharp
// Returning data:
public sealed record Create{Aggregate}Command(...)
    : ICommand<Create{Aggregate}Response>;

// Void operations (Update, Complete, Cancel, etc.):
public sealed record Update{Aggregate}Command(...)
    : ICommand<Unit>;
```

**Query definition:**
```csharp
public sealed record Get{Aggregate}ByIdQuery(Guid {AggregateId})
    : IQuery<{Aggregate}Dto>;
```

**Handler injection (endpoint):**
```csharp
public sealed class Create{Aggregate}Endpoint : IEndpoint
{
    private readonly ICommandHandler<Create{Aggregate}Command, Create{Aggregate}Response> _handler;
    // injected directly — no mediator, no bus, no magic
}
```

**Error handling:** All handlers return `Result<T>` — NO exceptions for business logic

**Pattern References:**
- See `.github/instructions/cqrs-implementation.instructions.md` — ICommand\<TResponse\>, handler rules, FORBIDDEN patterns
- See `.github/instructions/error-recovery.instructions.md` — Result\<T\> pattern
- See `.github/instructions/application-layer.instructions.md` — application layer structure

### 4.5 Multi-Tenancy Strategy [MANDATORY]

**Strategy:** Tenant-per-row isolation within shared schemas

**Rules:**
- Every DocumentStore query MUST include a `TenantId` filter
- TenantId is NEVER hardcoded — always injected via `ICurrentTenant`
- `ICurrentTenant` is resolved from JWT claims in the HTTP request pipeline

**Implementation:**
```csharp
// ✅ CORRECT — always filter by tenant
var events = await _context.DomainEvents
    .Where(e => e.TenantId == _currentTenant.TenantId)
    .Where(e => e.PartitionKey == aggregateId)
    .ToListAsync(ct);
```

**v1.0 Scope:** Single tenant (`"default"`) — multi-tenant capability built in via architecture, not activated

**Pattern Reference:** See `.github/instructions/multi-tenancy.instructions.md` — ICurrentTenant rules, query filter requirements

### 4.6 Connection String Management [MANDATORY]

**Strategy:** .NET Aspire service references — NO hardcoded connection strings

**AppHost registration:**
```csharp
var postgres = builder.AddPostgres("postgres").AddDatabase("{product}-db");

builder.AddProject<Projects.{Company}_{Product}_Services_{Domain}>("{domain}-service")
```

**Service consumption (Program.cs):**

**Development secrets:** Managed via `dotnet user-secrets` — NEVER committed to source control

**Pattern Reference:** See `.github/instructions/aspire-orchestration.instructions.md` — WithReference(), no hardcoded strings
